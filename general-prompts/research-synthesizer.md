# Research Synthesizer

Turn research into actionable briefings.

---

## Prompt

```
Create a research briefing on: [topic]

Include:
- key findings (5-10 main points)
- consensus (what experts agree on)
- disagreements (where opinions diverge)
- implications (what this means)
- action steps (what to do next)
```

---

## Example Usage

```
Create a research briefing on: the effectiveness of different code review practices

Include:
- key findings (5-10 main points)
- consensus (what experts agree on)
- disagreements (where opinions diverge)
- implications (what this means)
- action steps (what to do next)
```

---

## What You'll Get

**Key Findings**:
1. Async reviews > in-person for most changes
2. Smaller PRs reviewed faster and more thoroughly
3. Automated checks catch 40% of issues
4. Review time > 2 hours decreases quality
5. Pairing reduces review time by 60%

**Consensus**: 
- Code review improves quality
- Automation is valuable
- Smaller changes better

**Disagreements**:
- Pair programming vs async review
- Whether to require multiple reviewers
- When to use synchronous review

**Implications**:
- Invest in automation
- Break work into small PRs
- Time-box reviews

**Actions**:
1. Add linting and auto-formatting
2. Set PR size limits
3. Implement async-first review process

---

## When to Use

- Researching best practices
- Making informed decisions
- Need executive summary
- Evaluating tools or approaches
- Creating internal documentation
